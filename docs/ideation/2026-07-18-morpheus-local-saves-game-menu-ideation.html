What the sources actually say

Codebase Context

The TypeScript port already has an IndexedDB delta history, but gameplay mutations remain in Redux until a later scene transition commits them. The entry and active-head metadata are then written separately. A reload can therefore lose the latest meaningful action, and an interruption can expose a plausible-looking but incomplete checkpoint.

The persisted TypeScript shape also stops short of a full resume contract. It records a scene ID beside each delta, but hydration restores only gamestate values. Morpheus-javascript and Morpheus-C++ both establish a wider behavioral baseline: state values, current and previous scene context, panorama position, version handling, and player-facing save/load flows.

The current title offers only “Start a new game,” and that path clears persisted progress before the intro. The current top-left gear opens debug settings. Legacy JavaScript proves a top-left gear/menu button and a conventional menu; C++ proves a wheel cursor and dedicated Help-return semantics. The user calls the desired control the navigation wheel. Nothing in the evidence justifies turning the menu itself into radial spokes.

Core product judgment: use one automatic, durable “continue where I was” session as the default. Restore explicit load/export/import as ownership and recovery tools, not as a chore the player must perform after every puzzle.

Idea 1

1. The Living Passport Checkpoint

Top pick Confidence 94% Complexity High

Treat the durable resume as one complete, schema-versioned session passport: gamestate, active scene, semantically relevant previous/return context, and panorama position. After a meaningful action settles—or a scene transition reaches a coherent boundary—construct and validate the passport, then atomically promote it as the one current resume. Reload sees the last fully committed story moment, never a half-transition.

Illustrative flow: if any stage fails, the prior committed passport remains authoritative.

Axis
Save trigger and durability semantics
Meeting test
An interrupted transition resumes either the old coherent scene or the fully ready new one.
Basis
direct: current commits save an entry and then separately write stackHead (packages/www/src/morpheus-app/store/slices/gamestateSlice.ts:51-59); transition commit begins before target activation (packages/www/src/app/scene/stage-shell.tsx:372-380); C++ writes version, panorama, current/previous scenes, and every state value (original_source_code/MorpheusWin/CommonSources/MorpheusSupport/CGameState.cpp:160-187).

Rationale: It combines the two load-bearing corrections—write atomically and save the full playable place—into one continuity contract. It is the foundation on which every other save/menu idea can remain small.

Downsides“Meaningful action” needs an exact boundary. Too broad means write churn; too narrow preserves the current loss window. The complete envelope also creates the largest compatibility surface of the set.

Idea 2

2. The Honest Load Gate and Quarantine Shelf

Confidence 96% Complexity Medium

Never hydrate persisted bytes directly into live play. First validate the envelope, migrate supported versions, verify that its scene can be resolved, and wait for the runtime’s safe scene-ready boundary before applying rotation and dependent state. If validation fails, preserve the original save as a recoverable quarantine condition instead of silently truncating history or starting over.

Axis
Snapshot fidelity and evolution
Meeting test
A malformed, future-version, or unavailable-scene save cannot partially mutate the game.
Basis
direct: current chain resolution stops at a missing entry and returns accumulated values (packages/www/src/morpheus-app/storage/gamestateStorage.ts:92-109); hydration has no local recovery gate (packages/www/src/morpheus-app/hooks/useGamestateHistory.ts:19-31); C++ rejects incompatible major versions, bounds-checks values, and warns on missing scenes (original_source_code/MorpheusWin/CommonSources/MorpheusSupport/CGameState.cpp:60-129).

Rationale: Browser data outlives deployments. A truthful failure state protects both future engine evolution and the player’s trust far better than a green “Continue” button that lands in nonsense.

DownsidesMigration and quarantine can become a mini support product if allowed to sprawl. Keep the initial recovery vocabulary bounded: valid, migratable, preserved-but-unloadable, or empty.

Idea 3

3. The Navigation-Wheel Button as a Modal Input Boundary

Confidence 91% Complexity Medium

Restore the top-left navigation-wheel icon as a button that opens the conventional in-game menu. While the menu is open, it owns pointer, keyboard, touch, and focus; panorama drags, special-scene sliders, hotspots, and timed world interaction cannot also react. Closing by Resume, Escape, or the button restores focus and exactly one game-input authority.

This deliberately does not propose a radial-spoke menu. The user names the icon as the navigation wheel; the repository proves a top-left gear button, a conventional menu container, and a separate wheel cursor.

Axis
Navigation-wheel menu and input ownership
Meeting test
Dragging or pressing keys over an open menu cannot rotate the panorama or move a scene control.
Basis
direct: the user explicitly requests the top-left navigation-wheel icon; legacy MenuButton uses the gear asset and a 10px stage inset (upstream/master:packages/morpheus/client/js/morpheus/game/components/MenuButton.tsx:1-32); legacy Game conditionally renders Menu/Settings/SaveList (upstream/master:packages/morpheus/client/js/morpheus/game/containers/Game.tsx:11-53); current pointer handling has no cited menu-open gate (packages/www/src/morpheus-app/hooks/useInputHandler.ts:500-573).

Rationale: The menu is not decorative chrome. In pointer-sensitive Morpheus scenes, it is a mode boundary; treating it that way prevents exactly the kind of cross-input weirdness already exposed by slider work.

DownsidesA blunt global “paused” flag could suppress valid media or restoration work. The menu-open state needs one explicit owner and a carefully defined boundary between world input, media playback, and persistence.

Idea 4

4. Continue and New Game as Deliberate Choices

Confidence 97% Complexity Medium

Make the title honest about the local relationship. Show Continue only when the load gate confirms a valid resume, and take the player back to that session. Keep New Game as a separate path, with an explicit warning when it will replace durable progress. With no valid save, New Game remains the natural primary action and the established intro path stays intact.

Axis
Save UX and player ownership
Meeting test
Continue never lies, and New Game cannot silently clear a valid local resume.
Basis
direct: the current title exposes only Start a new game (packages/www/src/app/client.tsx:101-108) and its start path clears persistence (packages/www/src/app/client.tsx:39-76); legacy Continue loads the browser save only when one exists, while New deletes it (upstream/master:www/src/morpheus/title/actions.js:110-121).

Rationale: Continue is a promise about data validity; New Game is a destructive choice only when a valid story already exists. Making those semantics visible removes the title screen as a progress-loss footgun.

DownsidesResume validation is asynchronous, so the title needs a brief checking state rather than flashing the wrong action. The intro/new-game timing should remain behaviorally unchanged.

Idea 5

5. The Living Resume Card and Portable Memory

Confidence 93% Complexity Medium

Give the one local resume a compact player-facing card: where the story is, when it became durable, and whether it is valid or recoverable. Restore JSON export/import as the portability seam for browser-local ownership. Import validates into staging and preserves the current resume until the incoming session proves loadable.

Axis
Save UX and player ownership
Meeting test
A player can identify, export, and safely import the one resume without an account or silent overwrite.
Basis
direct: current persistence is one origin-local IndexedDB history (packages/www/src/morpheus-app/storage/gamestateStorage.ts:4-27, packages/www/src/morpheus-app/storage/types.ts:3-14); legacy browser save includes state, scene context, save ID, and rotation (upstream/master:www/src/morpheus/game/actions.ts:504-527); legacy file load/download uses morpheus.save.json and exposes a load-error path (upstream/master:www/src/morpheus/game/actions.ts:530-585).

Rationale: Local storage is convenient but origin-bound. One living resume plus a portable file restores meaningful player ownership without dragging accounts, cloud conflict resolution, or a slot-management spreadsheet into the game.

DownsidesImported files are untrusted input. Portability depends on the same versioning and quarantine discipline as reload, and the UI must avoid implying cloud-grade durability.

Idea 6

6. The Detour-Safe Chronicle

Confidence 95% Complexity Medium

Keep two kinds of memory deliberately separate. The validated passport is durable gameplay truth; Help and other menu detours receive a transient return context. Returning from Help is an authored continuation, not browser Back and not an alternate save history. A reload during a detour can still recover the last durable gameplay session without confusing route history for story progress.

Axis
Snapshot fidelity and evolution
Meeting test
Help, refresh, and browser Back never manufacture or replace a gameplay checkpoint.
Basis
direct: current GoBack uses one previousSceneId and has no live ReturnFromHelp case (packages/www/src/morpheus-app/hotspot/handleHotspotAction.ts:95-289); C++ stores a dedicated return scene for normal-to-help transitions (original_source_code/MorpheusWin/CommonSources/MorpheusSupport/CScene.cpp:430-463) and resolves it through ReturnFromHelp (original_source_code/MorpheusWin/CommonSources/MorpheusCast/CHotSpot.cpp:711-729).

Rationale: The original runtime distinguishes story navigation from help detours. Preserving that distinction keeps Continue, menu return, and browser navigation from becoming three competing versions of “where am I?”

DownsidesThe transient return token must never become a second durable authority. Its behavior across refresh needs an explicit product choice even if its storage stays ephemeral.

What did not earn a separate direction

Rejection Summary

# Idea or variant Reason rejected
1 The Continuous Durable Thread Its legacy automatic-save basis was weak and “meaningful action” was underspecified; its useful continuity goal is absorbed into Idea 1.
2 The Atomic Checkpoint Ceremony Sound, but only half of the stronger Living Passport Checkpoint; atomicity without a complete session still resumes the wrong place.
3 The Complete Versioned Session Passport Sound, but merged into Idea 1 so snapshot fidelity and write durability are decided together.
4 Radial wheel / authored spoke menu Basis refuted: the sources prove a top-left button, conventional menu, and wheel cursor—not a radial-spoke interaction.
5 The Wheel of Trust Duplicated the menu boundary and save card while repeating the unsupported radial-wheel premise. Save status can remain a small menu/card detail.
6 Authored Wheel Intentions The JavaScript Help/spoke claims were weak. The source-backed C++ Help return grammar survives as Idea 6.
7 The Recovery Observatory Sound but overlapping. Its preservation/export behavior belongs inside the Honest Load Gate and Resume Card.
8 Named manual slots and cloud saves Scope and complexity exceed the user’s reload-survival need. The evidence supports one local resume plus portable file ownership first.
